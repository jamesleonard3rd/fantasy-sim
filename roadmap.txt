================================================================================
FANTASY-SIM — WORLD SIMULATION ROADMAP
================================================================================

This file captures the design intent for the world simulation, the performance
reasoning behind it, and the concrete steps to build it. It is written so that
a fresh contributor (or AI context) can pick up the work without re-deriving
the architecture.

--------------------------------------------------------------------------------
1. PROJECT SHAPE
--------------------------------------------------------------------------------

Goal: a persistent fantasy world inhabited by hundreds-to-thousands of NPCs.
Simulation runs while a player/session is active. The world does NOT need to
advance while nobody is playing.

Three responsibilities must stay separate:

  * Runtime simulation (this Python codebase now; future server process):
      - Owns authoritative world state while a game session is active.
      - Runs coarse ticks for regions the player is NOT currently in.
      - Runs in memory; PostgreSQL is not the live simulation engine.

  * Live region (separate Unreal C++/Blueprint project):
      - Owns the single region the player is currently inside.
      - Runs full real-time simulation (frame-rate AI, combat, pathing).
      - Hydrates from the runtime/server on entry; sends meaningful changes
        back to the authoritative runtime.
      - In a multiplayer future, Unreal clients are presentation + input, not
        permanent truth.

  * Persistence (PostgreSQL for this prototype):
      - Stores saved world state between sessions.
      - Supports temporary dashboard/debug views while systems are being built.
      - Provides a migration path to future server-backed multiplayer.
      - Should be read/written at load/save/checkpoint boundaries, not per
        frame and not as the source of moment-to-moment gameplay state.

Schema: ECS-style. `entities` is the spine; everything else (traits,
factions, items, relationships, positions, zones, abilities, schools) is a
component table keyed by entity_id. See `backend/src/rts_world/db/schema.sql`.

--------------------------------------------------------------------------------
2. SCALE TARGETS
--------------------------------------------------------------------------------

  * World state: 2,000–5,000 entities total, partitioned into 20–30 regions.
    Average ~100–250 entities per region. Hard cap on region size is ~300
    entities to keep coarse tick cost predictable (see §4).
  * Live region (Unreal): typically 50–100 entities, peak 150–200.
  * Coarse unloaded-region tick interval while a session is active: 180 s
    target by default (configurable per region via
    `regions.tick_interval_seconds`). No ticks are required while no player
    session is running.
  * Per-entity logic is "heavy": each NPC scores multiple goals, considers
    interactions with other entities in its region, and applies probabilistic
    outcomes.

--------------------------------------------------------------------------------
3. PERFORMANCE ANALYSIS — WHY PYTHON IS FINE FOR COARSE SIMULATION
--------------------------------------------------------------------------------

Worry: "Will Python be too slow?"
Short answer: no, by 10–50x margin, IF the architecture below is followed.

Per-entity Python cost (CPython, dict-based state, utility-AI style scoring,
~50-neighbor interaction sweep): ~0.1–0.5 ms.
Per-region cost (250 entities, heavy sim): ~50–200 ms of CPU. If a tick
loads/writes through Postgres directly, add roughly ~150–300 ms of DB I/O.
Longer term, session-owned in-memory state should remove most DB I/O from the
steady-state tick path.

Scheduling while a player/session is active: staggered, not bursty.
  * 30 regions / 180 s budget = one region tick every ~6 seconds.
  * Per tick budget: 6 s wall clock. Real cost: far below that at target
    scale.
  * This means at any random instant the coarse sim is most likely idle, which
    is what makes co-locating with Unreal painless.

Where Python WOULD start to hurt (none of which apply at the target scale):
  * Region size > 500–1000 (the O(N^2) interaction term explodes).
  * Tick interval < 10 s combined with heavy logic.
  * Total entities > ~50,000.
  * Sub-millisecond real-time loops (that's Unreal's job, not Python's).

Escape hatches if any of the above happens later, in increasing cost:
  1. numpy / polars for any vectorizable scoring or spatial math.
  2. Spatial bucketing within regions to cap neighbor checks.
  3. Numba on the hot inner loop.
  4. Move only the inner interaction loop to Rust via PyO3.
  5. Last resort: rewrite the sim layer in another language. We don't
     expect to ever need this.

--------------------------------------------------------------------------------
4. CO-LOCATING WITH UNREAL — STUTTER-FREE COEXISTENCE
--------------------------------------------------------------------------------

For the single-player prototype, Unreal, the Python runtime, and Postgres may
all live on the player's machine. With sane defaults this is invisible from
Unreal's frame budget, but four specific footguns must be avoided:

  (a) Postgres fsync vs Unreal asset streaming on the same disk.
      Mitigations: require SSD; set `synchronous_commit = off` for local dev
      if acceptable; batch save/checkpoint writes. Do not commit per entity.

  (b) Lock contention from long Python writes blocking reads.
      Mitigations: keep transactions short; prefer checkpoint/delta writes;
      row-level MVCC handles the rest; never take table locks.

  (c) UNREAL POLLING POSTGRES ON THE GAME THREAD. This is the #1 cause of
      stutters in this kind of architecture and the single most important
      rule.
      Mitigations:
        - Unreal hydrates the live region from the runtime/server or
          persistence layer at boundaries. Never per-frame.
        - Any write-back to persistence goes through a worker thread or server
          process, never the game thread.
        - For "what happened while I was elsewhere" use the world_events log
          (see §5), read once at region entry or region load.

  (d) Python sim hogging a CPU core Unreal wants.
      Mitigations: run coarse ticks on a bounded cadence; avoid per-frame
      Python work; run the Python process at BelowNormal priority for local
      single-player dev if needed.

Recommended Postgres tuning for the player's local machine (single-user dev):

    shared_buffers          = 256MB
    effective_cache_size    = 1GB
    synchronous_commit      = off
    wal_compression         = on
    max_connections         = 20

--------------------------------------------------------------------------------
5. ARCHITECTURAL SOLUTION
--------------------------------------------------------------------------------

5.1 Region-as-unit-of-work
    The unit of coarse simulation, persistence batching, and reasoning is the
    REGION. Current implementation supports one wide SELECT (load),
    pure-Python simulate(), one bulk UPDATE / event INSERT (write). No N+1
    queries, no per-entity transactions.

    Future runtime target:
        session starts -> load world/regions into memory
        while active   -> pure-Python/server simulation over memory
        checkpoint     -> bulk save deltas/snapshots to persistence
        session ends   -> final save, stop simulation

5.2 Session-owned scheduler
    The scheduler should be thought of as a game-session runtime component,
    not an always-on offline daemon. It runs while a player/world session is
    active, and it stops when the session stops.

    Current implementation:
    A single asyncio (or thread-based) loop with a heap of
    (next_due_at, region_id). Pop the soonest-due region, sleep until
    due, run its tick, push it back with `now + tick_interval_seconds`.
    Skip regions where `paused = TRUE`.

    This is still a useful harness for building systems. Later, replace or
    wrap it with a session/server loop that keeps active world state in
    memory and uses Postgres only for load/save/checkpoint.

5.3 Data-oriented systems, not OO entities
    Do NOT instantiate 250 Entity objects per tick. Keep entity rows as
    dicts (or numpy columns later, if needed). Process by system, not by
    entity:

        for region in due_regions:
            state = load_region_state(region)
            for system in [needs, goals, relationships, faction_politics]:
                events.extend(system.apply(state, rng))
            write_region_state(state, events)

    Each system is a small file under `sim/systems/`. Adding a behavior
    is "add a file + register it in tick.py". Pure functions over
    region state make them trivially unit-testable.

5.4 Event log, not just state overwrites
    Systems should APPEND to `world_events` for anything a player might want
    to see ("Skirmish at the river", "House Vaelori enrolled a new student",
    "Captain Kael's opinion of the Mage Tower fell to -42"). When the player
    enters a region, Unreal/runtime pulls events since `last_seen_at` and
    replays/summarizes them. This makes coarse unloaded-region simulation feel
    like a living world without requiring simulation while nobody is playing.

5.5 Authority handoff
    Region state has exactly one authoritative writer at a time.

    Current simple contract:
      * If `regions.paused = TRUE`, the DB-backed background tick path skips
        the region because the live runtime owns it.
      * If `regions.paused = FALSE`, the coarse sim may tick it.

    Future multiplayer-friendly contract:
      * Runtime/server owns truth while a session is active.
      * Unreal clients send commands/input and receive replicated state.
      * Persistence stores checkpoints/saves, not per-frame truth.
      * Replace or extend `paused` with fields such as `authority_mode`,
        `authority_owner`, and `authority_lease_until` if/when multiple
        workers or multiplayer sessions exist.

5.6 World clock
    Single-row `world_clock` table holds shared in-game time. The
    current scheduler bumps it. All systems read game-time from TickContext
    rather than `datetime.now()` so they can be tested deterministically.
    Longer term, the session/server runtime should own clock advancement and
    persist it at save/checkpoint boundaries.

5.7 Travel graph authoring
    Region travel is resolved from authored `travel_edges` plus implicit local
    movement inside city-like region groups. A region whose `type` is `castle`,
    `city`, `settlement`, or `town` is a local travel group root; descendants
    of that root can travel to each other at 0 ticks with `mode = local`.
    Region `key` is the stable globally-unique identity used by parents and
    `travel_edges`; region `name` is display text and may repeat under
    different parents (for example multiple cities can each have "Barracks").

    Authoring rule: split dense cities into child regions under the city/castle
    parent, but do NOT add all-to-all zero-cost edges between those children.
    Add `travel_edges` only for leaving/entering the city, roads between
    settlements, borders, ports, gates, or special routes, and point those
    edges at region keys rather than display names. The routing layer derives
    linear zero-cost local links from the parent hierarchy so large cities do
    not create quadratic graph edges.

6. DATABASE — CURRENT STATE
--------------------------------------------------------------------------------

Schema lives in `backend/src/rts_world/db/schema.sql`. Already in place:

  Lookup / static:
    races, subraces, traits, stats, trait_stat_modifiers, factions,
    items, abilities, trait_abilities, schools

  Entity components:
    entities, entity_traits, entity_factions, entity_items,
    entity_abilities, entity_schools, entity_zones, entity_positions,
    relationships, relationship_terms

  Simulation / persistence infrastructure:
    regions          -- id, name, type, parent_id, tick_interval_seconds,
                        last_tick_at, paused
    world_events     -- id, region_id, kind, subject_entity_id,
                        target_entity_id, significance, payload JSONB,
                        occurred_at
    world_clock      -- single-row saved game time
    entity_zones.region_id  -- nullable FK alongside legacy zone TEXT

Important framing:
    PostgreSQL is the current persistence/debug store. It is not intended to
    be queried from Unreal per frame, and it is not the long-term home of live
    runtime state. Keep gameplay rules in pure systems so persistence can be
    swapped, batched, or moved behind a server later.

  Indexes added:
    idx_entity_zones_region, idx_world_events_region_time,
    idx_regions_due (partial: WHERE paused = FALSE),
    idx_relationship_terms_pair, idx_relationship_terms_source

Seeding pipeline (`backend/src/rts_world/db/seed.py`):
    Reads JSON from `game_data/{races,factions,stats,traits,abilities,
    items,schools}/...` and inserts idempotently. Run via
    `python backend/scripts/init_db.py`. The old SQL-based
    `seed_data.sql` was deleted; do not reintroduce it.

7. PROPOSED MODULE LAYOUT
--------------------------------------------------------------------------------

    backend/src/rts_world/
        api/                # existing FastAPI read-only views
        db/                 # existing schema + seed pipeline
        sim/
            __init__.py
            regions.py      # repository: load/write region working set,
                            #   query due regions, set paused
            tick.py         # tick_region(region_id) orchestrator;
                            #   pure-Python simulate() called between
                            #   load and write
            scheduler.py    # current heap-based active-session harness
            runner.py       # CLI entry for debug/dev harness
            events.py       # helpers for appending world_events
            clock.py        # read/advance world_clock
            systems/
                __init__.py
                needs.py        # hunger, fatigue, etc. drift
                goals.py        # goal scoring + selection
                relationships.py# opinion drift, gossip
                factions.py     # faction politics ticks
                # ... add a file per concern

Design rules:
  * Only `regions.py`, `events.py`, `clock.py` touch Postgres.
  * `tick.py` and everything under `systems/` are pure Python over
    in-memory state. Trivially unit-testable.
  * `runner.py` and `scheduler.py` are just loop wiring. No game logic lives
    there, and they can be replaced by a session/server runtime later.

9. SYSTEMS TO BUILD AFTER THE MVP (suggested order)
--------------------------------------------------------------------------------

In rough difficulty / dependency order:

  1. DONE: Relationship term dynamics.
     Relationship opinions are now derived from explainable
     `relationship_terms`. Temporary terms can decay toward 0 or expire,
     then cached `relationships.opinion` is recalculated and clamped.
     Validates: bulk UPDATE pattern and pure system unit tests.

  2. Needs (hunger, fatigue, mood) tied to traits.
     Validates: per-entity state mutation, system composition.

  3. Goals: each entity has 1–N active goals; tick scores them and picks
     a primary; primary goal modifies needs/relationships.
     Validates: multi-goal utility AI, the "heavy" per-entity work.

  4. Pairwise interactions: simple in-region neighbor sweep with bounded
     candidate count (cap N at ~30 per entity for safety, even in a
     250-entity region).
     Validates: O(N^2) cost stays bounded; spatial bucketing if needed.

  5. Faction politics: faction-vs-faction reputation ticks, opinion
     spillover from members.
     Validates: cross-system data flow.

  6. School enrollment / graduation lifecycle keyed off `world_clock`.
     Validates: time-based event scheduling.

  7. Birth / death / aging.
     Validates: entity lifecycle and FK cascade behavior.

After (6) and (7) unloaded regions can feel alive while the player is active
elsewhere, without requiring offline advancement while nobody is playing.

--------------------------------------------------------------------------------
10. RISKS / WATCHPOINTS
--------------------------------------------------------------------------------

  * O(N^2) explosion. If a region grows beyond ~300 entities, neighbor
    sweeps dominate. Two responses: cap region size at the design layer
    (split crowded regions), or add spatial bucketing in §9.4.

  * N+1 queries. The single most likely way to make the sim slow. All
    region loads must be wide SELECTs filtered by region_id. Code review
    rule: no SELECT inside a per-entity loop, ever.

  * Long transactions. One BEGIN/COMMIT per region tick. Never wrap the
    whole scheduler pass in a transaction.

  * Treating Postgres like runtime memory. Do not put per-frame gameplay or
    high-frequency entity movement through the DB. Load/save/checkpoint in
    batches; simulate in memory.

  * Schema drift between sim writes and API reads. The FastAPI layer in
    `api/main.py` already reads many of these tables. Any schema change
    must keep its reads working OR update them in the same commit.

  * Two writers. NEVER let both Unreal/live runtime and the coarse sim write
    to the same region simultaneously. The current `paused` flag is the
    contract; future multiplayer should replace it with explicit authority
    ownership/lease fields.

  * Unreal calling Postgres on the game thread. See §4(c). Worth a
    comment in the Unreal codebase too.

11.3 Conventions baked into the code (so they survive future edits)
  * Only sim/regions.py, sim/clock.py, sim/events.py touch psycopg.
    Everything under sim/systems/ and sim/tick.py is pure Python over
    in-memory state.
  * One BEGIN/COMMIT per region tick; `tick_region` opens its own
    connection unless the caller (tests) passes one in.
  * Systems get (state, ctx) and *return* PendingEvent rows; they never
    write to the DB and never call random.* directly (use ctx.rng so
    behaviour is deterministic given a seed).
  * Systems mutating per-entity rows MUST call
    state.mark_entity_dirty(...) / state.mark_relationship_dirty(...)
    or the change won't be flushed.
  * write_region_state already implements relationship and relationship-term
    flushes; the entities flush is a placeholder branch because no current
    system mutates entity rows. Wire it up when the first system needs it.

11.4 Relationship dynamics (current branch)
  * Added `relationship_terms` as the explainable source rows behind cached
    `relationships.opinion`.
  * Template relationships now seed matching terms, including kinship terms
    for parent/child/father/mother/sibling/spouse style relationships.
  * `relationship_dynamics` runs before the heartbeat system each tick. It
    decays temporary terms toward 0, zeroes expired terms, sums terms by
    directed subject/target pair, clamps the cached opinion to -100..100,
    and marks only changed rows dirty.
  * `regions.load_region_state` now loads relationship terms for in-region
    pairs, and `regions.write_region_state` bulk-updates dirty terms plus
    dirty cached opinions.
  * Unit coverage in `tests/test_relationship_system.py` verifies clamping,
    directional kinship, temporary decay, and expiry.

13. QUICK-START FOR A FRESH CONTEXT WINDOW
--------------------------------------------------------------------------------

If you're picking this up cold:

  1. Read §5 (architecture), §7 (module layout), and §11.3 (conventions).
     Everything else is justification.
  2. Apply schema + seed lookups + seed entities if needed:
        python backend/scripts/init_db.py
        python -m src.rts_world.db.seed_entities --random-count 50
            (run from backend/, or use the existing patterns)
  3. Create demo regions and assign entities:
        python backend/scripts/sim.py seed-regions
  4. See what's there:
        python backend/scripts/sim.py status
  5. Hand-tick one region (debug path):
        python backend/scripts/sim.py once --region-id 1
  6. Run the tests:
        pytest -v
  7. Run the current scheduler harness if you need to observe repeated ticks:
        python backend/scripts/sim.py forever

To add a new behaviour:
  1. Drop a file under backend/src/rts_world/sim/systems/. Export one
     function: `def my_system(state, ctx) -> list[PendingEvent]: ...`
  2. Mutate state in place; mark dirty rows; return events.
  3. Register in sim/systems/__init__.py default_systems().
  4. Add a unit test in tests/ that constructs a RegionState by hand.
     No DB needed — systems are pure.

--------------------------------------------------------------------------------
14. CURRENT PRODUCT DIRECTION
--------------------------------------------------------------------------------

Decision as of May 2026:

  * The world does not need to simulate while nobody is playing.
  * Both the live Unreal region simulation and coarse unloaded-region
    simulation should run only during an active player/world session.
  * The dashboard is temporary development tooling for visualizing data while
    systems are added.
  * Future multiplayer is possible, so avoid client-authoritative permanent
    state and avoid tying gameplay rules to local-only save formats.
  * All data must persist between sessions. A save file would work, but
    Postgres remains useful for the prototype because it already exists,
    supports the dashboard, and maps cleanly to a future server-backed model.

Recommended path:

  1. Keep building sim logic first under `sim/systems/`.
  2. Keep systems pure: no DB calls, no FastAPI calls, no Unreal assumptions.
  3. Use the current DB-backed tick path as a development harness.
  4. Later introduce a session/server runtime that loads world state into
     memory at session start, runs live + coarse simulation while active, and
     persists snapshots/deltas at checkpoint/end.
  5. For multiplayer, make the runtime/server authoritative. Unreal clients
     send commands/input and receive replicated results.
